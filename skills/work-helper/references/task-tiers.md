# 任务轻重分级标准

> 本文件定义重任务 / 轻任务的分界线。Agent 接到任务后按此分级，决定走完整流程还是简化流程。

---

## 分级标准

### 重任务（走完整流程）
满足以下**任意一条**即为重任务：

| 维度 | 阈值 |
|------|------|
| 影响文件数 | >3 个文件 |
| 新增代码行数 | >100 行 |
| 任务类型 | 新功能（new feature） |
| 任务类型 | 大重构（refactor，改公共接口/核心架构） |
| 涉及外部依赖 | 新增/升级/替换依赖 |
| 涉及数据库 | 改 schema / 迁移 |
| 涉及 CI/CD | 改构建/部署/工作流 |
| 涉及安全 | 改认证/授权/加密/密钥 |

### 轻任务（走简化流程）
满足以下**全部**即为轻任务：

| 维度 | 阈值 |
|------|------|
| 影响文件数 | ≤3 个文件 |
| 新增代码行数 | ≤100 行 |
| 任务类型 | bug fix / 小重构 / 补测试 |
| 不涉及外部依赖 | 不装/不升级依赖 |
| 不涉及数据库 | 不改 schema |
| 不涉及 CI/CD | 不改构建配置 |
| 不涉及安全 | 不改认证/授权/加密 |

---

## 边界情况

### 模糊任务
判断不了轻重时，**按重任务处理**（宁可多走流程，不可偷懒）。

### 用户明确指定
用户说"这是小改动""快速修一下"→ 可以按轻任务处理。
用户说"这是个大事""要认真做"→ 按重任务处理。

### 任务升级
轻任务执行中发现规模超出预期（比如 bug fix 翻出来要改 5 个文件），**立即升级为重任务**：
1. 停下来
2. 补完整 spec
3. 补 brainstorm
4. 继续走重任务流程

---

## 不同等级的流程差异

| 步骤 | 重任务 | 轻任务 |
|------|--------|--------|
| Spec | 完整 OpenSpec（proposal + tasks + delta-spec） | 简版 proposal（<20 行） |
| Brainstorm | 强制（3 种方案对比 + 风险点） | Agent 自决 |
| TDD | 完整 RED-GREEN-REFACTOR | 简化（bug fix 必须先写复现测试） |
| 子代理审查 | 强制 | 强制（不能省） |
| Commit | Conventional Commits | Conventional Commits |

**子代理审查不能省**——无论轻重任务都必须开子代理审查。

## 例子

### 重任务例子
- "加一个用户登录功能" → 新功能，重任务
- "把整个 auth 模块从 session 改成 JWT" → 大重构，重任务
- "加一个支付接口" → 新功能 + 涉及安全，重任务
- "升级 React 从 17 到 18" → 升级依赖，重任务

### 轻任务例子
- "这个函数报 NoneType 错误，修一下" → bug fix，轻任务（前提是改动小）
- "这个变量名拼错了" → 小重构，轻任务
- "给 utils.py 加几个测试" → 补测试，轻任务
- "把这段代码的 4 空格缩进改成 2 空格" → 纯样式，轻任务（甚至不走 TDD）
