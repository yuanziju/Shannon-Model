# 子代理审查 Checklist

> 本文件定义代码完工后的子代理审查流程。**所有代码任务都必须经过子代理审查才能交付。**

---

## 审查流程

### 1. 开子代理

主代理完成代码实现后，**必须**开一个新的子代理审查。

#### 子代理提示词模板（严格约束，不能放飞）

```
你是代码审查子代理。你的任务是审查以下代码改动，严格按照 checklist 逐项检查。

【约束】
1. 只审查，不修改代码
2. 不能开新的子代理（禁止递归）
3. 不能扩展审查范围（只看本次改动的文件）
4. 不能添加"建议"（除非是 must-fix 问题）
5. 必须按 checklist 逐项回答 PASS/FAIL，不能跳过

【改动文件】
<列出本次改动的文件路径>

【Spec】
<附上 proposal.md 内容>

【Checklist】
<见下方完整 checklist>

【输出格式】
对每一项回答：
- PASS: <说明>
- FAIL: <问题描述 + 必须修复的原因>

最后给出总结：MUST_FIX 数量 / NICE_TO_HAVE 数量 / 是否可以交付
```

### 2. 子代理不能开子代理（防递归）

- 子代理提示词里**明确禁止**开新子代理
- 子代理只能用 Read / Grep / Glob 等只读工具
- 子代理不能用 Write / Edit / RunCommand 等修改工具

### 3. 并行任务的验收

如果主代理并行开了多个子代理执行不同任务：
- **不能用执行任务的子代理自己验收自己的产出**
- 主代理必须**另开一个独立子代理**做验收
- 验收子代理审查所有并行子代理的产出

```
主代理
  ├── 子代理A（执行任务1）
  ├── 子代理B（执行任务2）
  ├── 子代理C（执行任务3）
  └── 验收子代理（审查 A/B/C 的产出，独立于 A/B/C）
```

### 4. 审查结果处理

- 子代理报 MUST_FIX → 主代理修复 → 再开新子代理复审
- 子代理报可以交付 → 进入 commit 步骤
- 子代理报 NICE_TO_HAVE → 主代理自决是否修复

---

## 完整 Checklist

子代理必须逐项检查：

### A. 防偷懒审查（最重要）

- [ ] **A1**：有没有 `pass` / `return None` / `...` 当实现？
- [ ] **A2**：有没有硬编码假数据冒充真实数据？
- [ ] **A3**：对照 spec，所有需求都实现了吗？
- [ ] **A4**：所有 TODO/FIXME 都有原因 + 计划 + 临时方案吗？
- [ ] **A5**：注释里的"应该做"都做了吗？
- [ ] **A6**：抽象层下面有真实实现吗？

### B. 功能正确性

- [ ] **B1**：实现符合 spec 描述的行为吗？
- [ ] **B2**：边界条件处理了吗（空值/负数/零/超长/非法输入）？
- [ ] **B3**：错误路径处理了吗（异常抛出/错误返回）？
- [ ] **B4**：happy path 和 sad path 都测了吗？

### C. 安全审查

- [ ] **C1**：有没有 SQL 拼接（应该用参数化查询）？
- [ ] **C2**：有没有硬编码密钥/密码/Token？
- [ ] **C3**：有没有命令注入风险（os.system / subprocess + shell=True）？
- [ ] **C4**：用户输入都校验了吗？
- [ ] **C5**：有没有越权风险（缺少权限检查）？

### D. 资源管理

- [ ] **D1**：文件/连接/流都关闭了吗（try-finally 或 with）？
- [ ] **D2**：线程池/进程池都 shutdown 了吗？
- [ ] **D3**：有没有内存泄漏风险（循环引用/全局变量累积）？
- [ ] **D4**：临时文件都清理了吗？

### E. 代码质量

- [ ] **E1**：命名风格统一吗（snake_case / camelCase 不混用）？
- [ ] **E2**：函数长度合理吗（>50 行考虑拆分）？
- [ ] **E3**：嵌套深度合理吗（>3 层考虑拍平）？
- [ ] **E4**：有重复代码吗（>3 处考虑抽函数）？
- [ ] **E5**：幻觉 API（不存在的库/方法）？

### F. 测试质量

- [ ] **F1**：测试名描述行为还是实现？
- [ ] **F2**：每个测试都有有意义的 assert？
- [ ] **F3**：测了错误路径，不只测 happy path？
- [ ] **F4**：断言具体值，不是 `is not None` / `True`？
- [ ] **F5**：测试独立吗（不依赖其他测试）？
- [ ] **F6**：测试确定性吗（不依赖时间/网络/随机）？
- [ ] **F7**：关键路径都有测试吗？

### G. Spec 一致性

- [ ] **G1**：实现符合 proposal.md 的目标吗？
- [ ] **G2**：tasks.md 列的任务都完成了吗？
- [ ] **G3**：非目标（明确不做的）真的没做吗？
- [ ] **G4**：delta-spec 的 ADDED/MODIFIED/REMOVED/RENAMED 都对得上吗？

### H. Commit 准备

- [ ] **H1**：commit message 符合 Conventional Commits 吗？
- [ ] **H2**：改动范围和 commit type 匹配吗（feat/fix/refactor/test/docs）？
- [ ] **H3**：scope 选对了吗？

## 子代理输出格式

```markdown
# 代码审查报告

## 改动文件
- src/auth/login.py
- tests/test_login.py

## Checklist 结果

### A. 防偷懒审查
- A1: PASS - 无空实现
- A2: FAIL - login.py:42 硬编码了 token = "fake-token"，必须修复
- A3: PASS - spec 要求都实现
- ...

### B. 功能正确性
- B1: PASS
- B2: FAIL - 没处理 password 为 None 的情况
- ...

## 总结
- MUST_FIX: 3 (A2, B2, F3)
- NICE_TO_HAVE: 1 (E3)
- 是否可交付: 否（有 3 个 MUST_FIX）
```

---

## 主代理收到审查结果后

### MUST_FIX 处理
1. 逐项修复
2. 修复完后**再开一个新子代理**复审（不能让原子代理复审，避免"自己审自己"的偏差）

### NICE_TO_HAVE 处理
- 简单的顺手修
- 复杂的留 TODO，记录在 spec 里

### 反复修复
- 同一个问题修复 2 次还是 FAIL → 停下来用 AUQ 问用户（见停问时机）
- 可能是理解偏差，需要和用户对齐

## 子代理提示词约束总结

| 约束 | 说明 |
|------|------|
| 只审查不修改 | 子代理不能用 Write/Edit/RunCommand |
| 禁止递归 | 子代理不能再开子代理 |
| 禁止扩展范围 | 只审查本次改动的文件，不扩散 |
| 禁止建议 | 只报 PASS/FAIL，不给"建议"（除非 MUST_FIX） |
| 必须逐项回答 | 不能跳过 checklist 任何一项 |
| 输出格式固定 | 必须按模板输出，方便主代理解析 |
